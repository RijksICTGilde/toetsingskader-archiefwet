Deel 1: Wat is GitHub en hoe werkt versiebeheer?

De teksten van het toetsingskader staan op GitHub: een website waar bestanden samen met hun volledige geschiedenis worden bewaard. Elke wijziging wordt vastgelegd: wie, wanneer, wat en waarom. Er gaat nooit iets verloren en alles is terug te draaien.

Vijf begrippen

BegripBetekenis
Repository De map met alle bestanden van het toetsingskader (plus de hele geschiedenis).
Commit Eén opgeslagen wijziging, met een korte omschrijving van wat je deed.
Pull request (PR) Een voorstel: "ik wil deze wijziging doorvoeren". Anderen kunnen het bekijken vóór het echt gebeurt.
Check Een automatische controle die elk voorstel naleest op fouten in de opbouw (kop vergeten, voetnoot zonder bron, …).
Release Een gepubliceerde, genummerde momentopname van de site (bijvoorbeeld v0.1.0): een set doorgevoerde wijzigingen samen onder één versienummer.

De route van elke wijziging

  1. Jij bewerkt een pagina op github.com.
  2. GitHub maakt er een voorstel (pull request) van.
  3. De automatische controle leest het na: groen vinkje (in orde) of rood kruis (er klopt iets niet, met uitleg in het Nederlands, inclusief regelnummer).
  4. Een collega keurt goed.
  5. Daarna pas komt de wijziging op de website.

Losse wijzigingen worden van tijd tot tijd gebundeld tot een release: een genummerde versie (bijvoorbeeld v0.1.0) die vastlegt hoe de site er op dat moment uitzag. Zo blijft navolgbaar wat er in welke versie veranderde. Hoe je releases leest en zelf aanmaakt, staat in 2.7.

Je kunt niets kapotmaken. De website verandert nooit direct door wat jij typt. Elke stap is omkeerbaar, en de controle plus een collega kijken altijd mee voordat iets live gaat.

Deel 2: Handleiding (naslag)

2.1 Een pagina vinden en bewerken

  1. Ga naar github.com/RijksICTGilde/toetsingskader-archiefwet.
  2. Klik op de map content, dan normen.
  3. Klik op de norm die je wilt aanpassen (bijv. 02-overzicht.md).
  4. Klik rechtsboven op het potlood-icoon ("Edit this file").

Wat je dan ziet: bovenin, tussen twee ----strepen, staan de titel, de kern en de synoniemen van de norm. Daaronder staat de gewone tekst van de pagina, met koppen die beginnen met ##.

Tip: met de tab "Preview" bovenin zie je direct hoe je tekst eruit komt te zien.

2.2 De opbouw van een pagina

KopBetekenis

## Normuitleguitwerking (verplicht), met daarbinnen:
### <thema>subkop, bijvoorbeeld "Besturing"

Voorbeeldfragment:

## Normuitleg

### Besturing

#### Voorschrift

De Inspectie toetst of regels voor het archiefbeheer zijn vastgesteld.

#### Criteria

- De vastgestelde beheerregels zijn actueel.
- De vastgestelde beheerregels zijn volledig.

2.3 Bronnen toevoegen (voetnoten)

Zet [^naam] direct achter het woord dat je onderbouwt en zet de bron eronder. De nummering en de bronnenlijst onderaan de pagina gaan vanzelf. Die hoef je nooit zelf bij te houden.

Een document moet in beheer zijn.[^aw-4-1-lid-1]

[^aw-4-1-lid-1]: Aw, artikel 4.1, lid 1. [Bekijk bron](https://zoek.officielebekendmakingen.nl/kst-35968-2.html)
  • Zelfde bron nog een keer gebruiken? Typ alleen opnieuw [^aw-4-1-lid-1]. De regel eronder hoeft niet herhaald.
  • Bron zonder link? Mag: laat het [Bekijk bron](…)-deel weg.
  • Namen: kleine letters, cijfers en koppeltekens, bijvoorbeeld [^kamerstuk-35968].

2.4 Wijziging indienen

  1. Klik op de groene knop "Commit changes…".
  2. Vul een korte omschrijving in (bijv. "Toelichting norm 2 aangevuld").
  3. Kies "Create a new branch for this commit and start a pull request".
  4. Klik "Propose changes" en daarna "Create pull request".

2.5 De automatische controle lezen

Onderaan je pull request verschijnt na korte tijd het resultaat:

  • Groen vinkje: alles in orde. Wachten op goedkeuring van een collega.
  • Rood kruis: klik op "Details". Je ziet dan een melding zoals: 
    04-metadatering.md:23: Gebruik '#### Criteria' (meervoud) in plaats van '#### Criterium'
    Dat betekent: bestand 04-metadatering.md, regel 23. Open het bestand opnieuw via het potlood-icoon in dezelfde pull request (tabblad "Files changed" → potlood), herstel het, en commit opnieuw. De controle draait dan vanzelf opnieuw.

2.7 Releases en versies

Een release is een genummerde momentopname van de site. Waar een pull request één wijziging doorvoert, bundelt een release een set doorgevoerde wijzigingen onder één versienummer, zodat naderhand precies te zien is wat er in welke versie zat.

Het versienummer heeft drie delen, bijvoorbeeld v0.1.0:

  • het laatste getal loopt op bij kleine correcties;
  • het middelste getal bij nieuwe of uitgebreide inhoud;
  • het eerste getal bij grote, ingrijpende wijzigingen.

Wat er per versie veranderde, staat in CHANGELOG.md in de repository en op de Releases-pagina van GitHub (github.com/RijksICTGilde/toetsingskader-archiefwet/releases).

Zelf een release aanmaken (in de GitHub UI):

  1. Open de repository op GitHub en klik rechts op "Releases", daarna op "Draft a new release".
  2. Bij "Choose a tag": typ het nieuwe versienummer (bijvoorbeeld v0.1.0) en kies "Create new tag on publish". Laat het doel op main staan.
  3. Vul een titel in (bijvoorbeeld v0.1.0) en beschrijf in het tekstvak kort de wijzigingen. Met de knop "Generate release notes" vult GitHub automatisch de doorgevoerde pull requests sinds de vorige release in; vul ze zo nodig aan vanuit CHANGELOG.md.
  4. Klik "Publish release".
Tip: werk het versienummer in CHANGELOG.md (en zo nodig publiccode.yml) bij in een gewone pull request voordat je de release publiceert, zodat de release en de documentatie hetzelfde versienummer noemen.

Deel 3: Oefeningen

Vier opdrachten. De trainer begeleidt en ruimt na afloop de oefen-voorstellen op (sluiten zonder doorvoeren). De website blijft dus ongewijzigd.

Oefening A: Tekst wijzigen

Doel: een pagina openen, bewerken en je wijziging in de preview zien.

  1. Open content/normen/02-overzicht.md en klik het potlood-icoon.
  2. Vervang in de sectie ## Toelichting de tekst [PLACEHOLDER: vul toelichting in] door één eigen zin.
  3. Bekijk het resultaat via de tab "Preview".

Klaar wanneer: je ziet jouw zin netjes opgemaakt in de preview.

Oefening B: Bron toevoegen

Doel: een voetnoot maken.

  1. Zet in dezelfde bewerking achter jouw zin: [^mijn-bron].
  2. Zet er een lege regel onder en daarna: 
    [^mijn-bron]: Eigen aantekening als oefening.
  3. Bekijk de preview.

Klaar wanneer: de preview toont een voetnootnummer achter je zin en je aantekening onderaan.

Oefening C: Voorstel indienen + controle lezen

Doel: de hele route doorlopen, inclusief een bewuste fout herstellen.

  1. Verwijder eerst expres de definitieregel ([^mijn-bron]: …) uit je tekst.
  2. Klik "Commit changes…", kies "Create a new branch", maak de pull request aan.
  3. Wacht op het rode kruis, klik "Details" en lees de melding.
  4. Herstel de fout in dezelfde pull request (tabblad "Files changed" → potlood) en commit opnieuw.
  5. Wacht tot de check groen wordt en wijs rechts een reviewer aan (Reviewers → kies je collega of de trainer).

Klaar wanneer: groene check én een toegewezen reviewer.

Oefening D: Een release lezen

Doel: de huidige versie vinden en de bijbehorende release-notes lezen. Je maakt hierbij niets aan; je leest alleen.

  1. Ga naar de repository op GitHub en klik rechts op "Releases".
  2. Open de bovenste (meest recente) release en lees de notes: welk versienummer is het, en wat is er in die versie veranderd?
  3. Open CHANGELOG.md in de repository en zoek hetzelfde versienummer op. Komt de beschrijving overeen?

Klaar wanneer: je het huidige versienummer kunt noemen en in eigen woorden kunt zeggen wat er in die release zat.

Vragen na de training? Open een issue op GitHub (github.com/RijksICTGilde/toetsingskader-archiefwet/issues) of mail de beheerder. Naslag: zie ook CONTRIBUTING.md in de repository.
